This document provides an updated summary of the current state of the SimpleECommerce workspace across both frontend and backend as of March 9, 2026.

## Current Architecture

### Workspace Structure

The project is currently split into two main application areas plus supporting documentation and local infrastructure:

```text
SimpleECommerce/
├── simple-e-commerce-frontend/
├── SimpleECommerceBackend/
├── docs/
└── docker-compose.yml
```

### Technology Stack

- **Frontend**: Angular 21 standalone application
- **Frontend Styling**: Tailwind CSS v4 imported through global styles
- **Frontend Serving**: Nginx in Docker for built assets
- **Backend Framework**: ASP.NET Core (.NET 10)
- **Backend Architecture**: Clean Architecture with Api, Application, Infrastructure, and Domain projects
- **Authentication**: Keycloak with OAuth2/OIDC
- **Database**: SQL Server for business data
- **Identity Provider**: Keycloak 24 with PostgreSQL
- **Mapping**: Mapster
- **Application Messaging**: MediatR
- **Testing**: xUnit, FluentAssertions, Moq, ASP.NET Core integration testing

## Authentication & Authorization State

### 1. Current Authentication Model

**Location**: `SimpleECommerceBackend/SimpleECommerceBackend.Api/Program.cs`

**Current Implementation**:

- Uses `AddKeycloakWebApiAuthentication`
- Configures role-based authorization policies for:
  - `RequireCustomerRole`
  - `RequireSellerRole`
  - `RequireAdminRole`

**Target flow**:

1. The Angular SPA redirects directly to Keycloak for login and registration
2. Keycloak returns the authorization response to the SPA
3. The SPA calls the backend with the access token
4. The backend validates the token and never owns the browser login flow

### 2. Auth Endpoints

**Location**: `SimpleECommerceBackend/SimpleECommerceBackend.Api/Controllers/AuthController.cs`

**Current Endpoints**:

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`

These endpoints are legacy or transitional only. The preferred solution is to remove interactive auth endpoints from the browser flow and let the SPA talk to Keycloak directly.

### 3. Keycloak Integration State

**Locations**:

- `SimpleECommerceBackend.Infrastructure/Services/Keycloak/KeycloakTokenService.cs`
- `SimpleECommerceBackend.Infrastructure/Services/Keycloak/KeycloakAdminService.cs`
- `SimpleECommerceBackend.Api/appsettings.Development.json`

**Current State**:

- Backend is configured against a local Keycloak realm running at `http://localhost:8080`
- The backend acts as a resource server for access tokens issued to the SPA
- Service-account access is only needed for backend-side admin automation or user provisioning
- The frontend is the only browser-facing OIDC client

### 4. Auth Flow Behavior

**Registration Flow**:

1. The SPA redirects to Keycloak for registration or sign-in
2. Keycloak handles authentication and token issuance
3. The SPA stores and refreshes tokens according to its client-side security policy
4. The backend accepts the access token and maps the local `UserProfile` on demand

**Login Flow**:

1. Validate the Keycloak bearer token
2. Map the `sub` claim to the local user profile
3. Apply backend authorization policies and business rules

**Refresh Flow**:

1. The SPA refreshes tokens directly with Keycloak
2. The backend does not proxy refresh tokens for browser users
3. Server-side automation can still use Keycloak service accounts when necessary
